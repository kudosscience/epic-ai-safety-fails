## Requirements

- Identify the exact request payload and schema mismatch causing the error.
- Confirm whether the error is caused by client input, reCAPTCHA token, or server validation.
- Define a minimal fix path without scope creep.

## Current Observations

- The server schema in [src/app/api/auth/signup/route.ts](../../../src/app/api/auth/signup/route.ts) requires `username` to match a lowercase-only pattern and rejects uppercase or hyphenated values.
- The client form in [src/components/signup-form.tsx](../../../src/components/signup-form.tsx) now normalizes `username` to lowercase; ensure the deployed build includes this change.
- The server schema requires `recaptchaToken` to be present and at least 10 characters; missing or blocked reCAPTCHA will surface as "Invalid signup payload" before CAPTCHA verification runs.
- The server schema requires `fullName` to be at least 2 characters and `password` to be at least 12 characters; the UI enforces both minimums in the signup form.
- The server schema requires `inviteCode` to be at least 8 characters; short placeholders will fail payload validation before invite lookup.

## Implementation Plan

1. Reproduce
   - Attempt signup in the test environment and capture the request body in DevTools.
   - Note all fields (`email`, `password`, `username`, `fullName`, `inviteCode`, `recaptchaToken`).
2. Validate schema alignment
   - Compare the payload against `signupRequestSchema` in [src/app/api/auth/signup/route.ts](../../../src/app/api/auth/signup/route.ts).
   - Check `email` format, `username` regex (lowercase only), invite code length/trim rules (min 8), `fullName` min length, and recaptcha token length.
3. Check reCAPTCHA flow
   - Ensure `recaptchaToken` is present and generated after the script is ready.
   - Confirm the expected action value matches `signup`.
4. Review server-side response
   - Inspect API response and logs for validation errors.
   - Confirm the error path (`Invalid signup payload`) is reached due to schema parse failures.
5. Remediation plan
   - If client formatting is wrong, normalize `username` to lowercase client-side and add inline validation messaging.
   - If invite code length causes payload rejection, add client validation and helper messaging for the 8-64 character range.
   - If schema is too strict for the UI, adjust schema with a documented rationale and keep server-side validation strict.
   - If reCAPTCHA is missing, verify site key configuration, script readiness, and add a clearer client error state.

## Manual Steps (Invite Codes)

1. Open your Supabase project for the environment you are testing.
2. Go to Table Editor and open the `research_invites` table.
3. Insert a new row with:
   - `code`: an 8-64 character value (uppercase or lowercase is fine; the server normalizes to uppercase).
   - `invited_by`: leave null if you do not have an inviter profile yet.
   - `used_by`, `used_at`, `revoked_at`: leave null.
4. Save the row and use the `code` value in the signup form.
5. If the table is missing, apply your schema migration or recreate the `research_invites` table before testing.
